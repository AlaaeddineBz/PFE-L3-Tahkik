\chapter{Implementation}

\section{Introduction}
This chapter presents the technical implementation of the Tahkik platform. We begin by introducing the languages and frameworks used across all system components, then walk through the realised interfaces of both the mobile application (\textit{Tahkik App}) and the web dashboard (\textit{Tahkik Dashboard}).

\begin{infobox}[title=Scope of this Chapter]
Two specialised components --- the \textbf{Go/Python inference server} and the \textbf{fine-tuned Whisper model} --- are described at a high level here and documented in depth in two companion technical reports that are appended to this dissertation:
\begin{itemize}[leftmargin=2em]
  \item \textit{Tahkik Inference Server --- Technical Report} (Go API gateway, FastAPI inference backend, faster-whisper / CTranslate2 runtime, WebSocket streaming protocol, Hugging Face Space deployment, performance \& security).
  \item \textit{AI-Powered Quranic Recitation Analysis --- Implementation Report} (Warsh dataset preparation, data quality assurance pipeline, Whisper fine-tuning runs, WER results, model publication on Hugging Face Hub).
\end{itemize}
This chapter therefore concentrates on the two client-side components that are not covered elsewhere: the Flutter mobile application and the React web dashboard.
\end{infobox}

\textbf{Flutter} is Google's open-source UI toolkit for building natively compiled applications for mobile, web, and desktop from a single codebase. \subsection{Backend API — Go}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.22\textwidth]{figures/tech/golang.png}
  \caption{Go logo}
  \label{fig:impl_go}
\end{figure}

\textbf{Go} (Golang) is a statically typed, compiled language developed by Google, known for its simplicity and high concurrency performance. The Tahkik Go server acts as a lightweight REST gateway in front of the Python inference backend: it accepts batch audio uploads on \texttt{POST /api/v1/evaluate}, validates them, forwards them to the inference service, and exposes a Bearer-token-gated \texttt{GET /api/v1/system-health} endpoint (plus its WebSocket variant \texttt{/api/v1/system-health/ws}) that the web dashboard polls for live host, container, and inference telemetry. A public \texttt{GET /health} liveness probe is also provided for Caddy and Docker healthchecks. The complementary real-time WebSocket streaming path (\texttt{/ws/stream}) is served by the FastAPI inference backend directly to the mobile client; the Go server is intentionally kept stateless and proxy-free for streaming workloads. The full server architecture, request lifecycle, and deployment procedure are detailed in the companion \textit{Tahkik Inference Server} report.

\textbf{Python} is the leading language for machine learning and audio processing. The production AI backend runs NVIDIA's \textbf{NeMo FastConformer Hybrid CTC/RNNT} model (\texttt{nvidia/stt\_ar\_fastconformer\_hybrid\_large\_pcd\_v1.0}, $\sim$115\,M parameters), achieving 6.55\,\% WER on the EveryAyah Quran benchmark. It is deployed as a Docker container on a DigitalOcean droplet, behind the Caddy reverse proxy. For local development, a legacy PyTorch server backed by the fine-tuned Whisper checkpoint (\texttt{benhadjermed/tahkik-small-warsh}) is retained. The full fine-tuning pipeline --- Warsh dataset alignment, quality-assurance scripts, multi-reciter training runs and WER analysis --- is documented in the companion \textit{AI-Powered Quranic Recitation Analysis} report.

\textbf{FastAPI} is a modern, high-performance Python web framework for building APIs with automatic OpenAPI documentation. It serves as the HTTP/WebSocket interface for the NeMo inference pipeline: it exposes \texttt{POST /evaluate} for batch audio (called by the Go gateway), \texttt{WS /ws/stream} for the real-time 8-second sliding-window streaming protocol consumed directly by the Flutter mobile client, and \texttt{GET /health} for liveness and metrics.

% ─── Practice ───────────────────────────────────────────────────────────────
\subsection{Practice Screen and Tajweed Feedback}

The Practice screen is the core feature of the app. The user selects a verse range, then a large \textbf{Microphone FAB} (\texttt{microphone\_fab.dart}) starts recording with the \texttt{record} package at 16~kHz PCM. A live \textbf{Waveform Visualiser} animates while audio chunks are streamed over WebSocket to the inference service. Partial transcriptions arrive every $\sim$1\,s and are overlaid in real time on the corresponding \textbf{Ayah Card}. A \textbf{Practice Theme Selector} lets the user switch between calm, focus and traditional colour palettes.

When recording stops, the final result --- transcription, confidence score, and per-word Tajweed annotations --- is rendered by the \texttt{tajweed\_feedback} module:

\begin{itemize}[leftmargin=2em]
  \item \textbf{Overall Score widget}: animated radial gauge showing the confidence percentage.
  \item \textbf{Tajweed Overlay}: highlights each word with a colour reflecting the detected error category (madd, ghunnah, qalqalah, etc.).
  \item \textbf{Error Detail Sheet}: a bottom sheet that, when a highlighted word is tapped, explains the rule, plays the correct pronunciation, and proposes a corrective exercise.
\end{itemize}

\section{Web Dashboard — Tahkik Dashboard}
\label{sec:impl_dashboard}

\begin{emeraldbox}[title=Web Dashboard]
The Tahkik Dashboard is a React~18 + TypeScript single-page application built with Vite, React Router v6 (data router), TanStack Query~v5 for server state, and Recharts for visualisations. It is the unified entry point for both administrators and teachers --- role-based routing redirects each authenticated user to the appropriate workspace based on their record in the \texttt{admins} table. The dashboard talks directly to Supabase for authentication, database reads/writes, and real-time subscriptions, and consults the Tahkik Go server (\texttt{VITE\_TAHKIK\_SERVER\_URL}) for live inference telemetry. Styling is built on CSS Modules with a tokenised light/dark theme, and the test suite uses Vitest + React Testing Library.
\end{emeraldbox}

% ─── Dashboard architecture & libraries ────────────────────────────────────

\subsection{Application Shell, Auth Guard and Error Boundary}

The router is built with React Router's \emph{data router} API. A top-level \texttt{ErrorBoundary} catches rendering and loader exceptions and surfaces a friendly retry screen, while \texttt{RequireAuth} wraps every protected route, subscribing to \texttt{AuthProvider} to enforce the following invariants:

\begin{itemize}[leftmargin=2em]
  \item Unauthenticated visitors are redirected to \texttt{/login}.
  \item Authenticated users without a matching row in the \texttt{admins} table are sent to \texttt{/unauthorized}.
  \item Visiting \texttt{/} redirects an admin to \texttt{/admin} and a teacher to \texttt{/teacher}, based on the \texttt{role} column.
\end{itemize}

\texttt{AuthProvider} listens to \texttt{supabase.auth.onAuthStateChange} and exposes \texttt{\{ user, role, isLoading \}} to the rest of the tree via React context, so every page consumes a single source of truth for authentication.

\subsection{Styling and Light/Dark Theme}

All visual tokens (colours, spacing, radii, typography, shadow, transition durations) are declared as CSS custom properties in \texttt{theme/tokens.css}. A \texttt{ThemeProvider} toggles a \texttt{data-theme} attribute on the document root between \texttt{light} and \texttt{dark}, and components consume the tokens through scoped CSS Modules. Skeleton loaders (\texttt{Skeleton}, \texttt{SkeletonPage}, \texttt{SkeletonShell}) reuse the same tokens so the loading state is visually consistent across pages.

\subsection{Data Layer --- TanStack Query Hooks}

Every page is hydrated through TanStack Query hooks split by domain:

\begin{itemize}[leftmargin=2em]
  \item \texttt{lib/queries/admin.ts}: \texttt{useAdminOverview}, \texttt{useAdminUsers}, \texttt{useAdminTeachers}, \texttt{useAdminSubscriptions}, \texttt{useAdminAnalytics}, etc.
  \item \texttt{lib/queries/teacher.ts}: \texttt{useTeacherOverview}, \texttt{useTeacherStudents}, \texttt{useStudentDetail}, \texttt{useRecitationQueue}.
  \item \texttt{lib/queries/messages.ts} and \texttt{useMessagesRealtime}: backed by a Supabase real-time channel for in-app chat.
  \item \texttt{lib/queries/certificates.ts} and \texttt{lib/queries/daily-verses.ts}: read hooks for the Ijaza and Daily Verse modules.
  \item \texttt{lib/mutations/}: write counterparts for teacher assignment, certificate issuance, and message sending, all using TanStack Query's \texttt{onSuccess} cache invalidation to keep the UI in sync.
\end{itemize}

\subsubsection{AI \& System Health}

This is the live operations console for the Tahkik AI pipeline. The dashboard polls the Go server's health endpoint and renders:
\begin{itemize}[leftmargin=2em]
  \item \textbf{Host metrics} --- CPU, memory, disk, and swap utilisation with critical/warning thresholds.
  \item \textbf{Inference latency} --- partial and final p95 latency in milliseconds.
  \item \textbf{Request stats} --- total and failed request counts with derived error-rate alerts.
  \item \textbf{Container health} --- Docker container status (running, unhealthy, exited).
  \item \textbf{Live alert strip} --- automatically raises a Sonner toast when a metric crosses the critical threshold.
\end{itemize}

\section{Conclusion}
This chapter presented the full technology stack powering the Tahkik platform and the realised interfaces of both the mobile application and the web dashboard. The Flutter app combines Riverpod, GoRouter, Dio, \texttt{web\_socket\_channel} and the custom KFGQPC Warsh font to deliver a complete learner journey from onboarding to subscription. The React dashboard layers TanStack Query, Recharts, CSS Modules and Sonner on top of Supabase to give administrators and teachers a unified, live, and accessible operations console. The Go gateway and the fine-tuned Whisper inference service form the AI backbone --- a backbone whose internals, training procedure, and deployment topology are detailed in the two companion technical reports appended to this dissertation.
