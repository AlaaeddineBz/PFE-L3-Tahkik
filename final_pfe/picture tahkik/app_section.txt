3.2.3 Backend API — Go
Figure 3.3: Go logo
Go(Golang) is a statically typed, compiled language developed by Google, known for its
simplicity and high concurrency performance. The Tahkik Go server acts as a lightweight
REST gateway in front of the Python inference backend: it accepts batch audio uploads on
POST /api/v1/evaluate , validates them, forwards them to the inference service, and ex-

for Caddy and Docker healthchecks. The complementary real-time WebSocket stream-
ing path ( /ws/stream ) is served by the FastAPI inference backend directly to the mobile
client; the Go server is intentionally kept stateless and proxy-free for streaming workloads.
The full server architecture, request lifecycle, and deployment procedure are detailed in
the companion Tahkik Inference Server report.

4. Introduction
This technical report documents the server-side infrastructure of the Tahkik project —
an AI-powered Quranic recitation analysis system specialising in the Warsh ‘an Naﬁ’
riwaya (Tariq al-Azraq). The server bridges the Flutter mobile application with the
speech-recognition model, accepts audio uploads (or live PCM streams), runs an optional
Tajweed (madd) analysis against a reference text, and returns structured transcription
and error results.
The production stack is composed of three containerised components running side-
by-side on a single Linux host: a Caddy reverse proxy (auto-TLS via Let’s
Encrypt), a Go HTTP API gateway, and a Python FastAPI inference server
backed by the NVIDIA NeMo FastConformer Hybrid CTC/Transducer model
(nvidia/stt_ar_fastconformer_hybrid_large_pcd_v1.0 ). A WebSocket stream-
ing endpoint ( /ws/stream ) delivers real-time partial transcription, and a separate
/api/v1/system-health endpoint (plus its WebSocket variant) exposes host metrics,
container status, and inference latency for the operations dashboard.
This report covers the architecture, component details, data pipeline, madd analyser, API
contracts, deployment procedure, security model, and operational concerns of the server.

7. Data Pipeline
7.1Audio Processing Flow
When an audio ﬁle arrives at the server, it passes through the following stages:
Phase 1 — Audio Reception
The Go server receives the multipart upload, validates the ﬁle extension and size (max
50 MB), reads the binary payload into memory, optionally extracts the reference_text
form ﬁeld, and forwards both to the Python backend.
Phase 2 — Audio Preprocessing
The Python backend writes the bytes to a temp ﬁle under /tmp , loads it via
librosa.load(sr=16000, mono=True) , and clamps the duration to the last 60 seconds.
Unlike Whisper, NeMo FastConformer handles long audio directly, so the legacy 30-
second chunking step is no longer required.
Phase 3 — Inference
The audio array is handed to model.transcribe(...) . When a reference_text
is present, the call is invoked with return_hypotheses=True, timestamps=True so
the wrapper can extract per-word and per-character timestamps from the returned
Hypothesis object.
Phase 4 — Optional Madd Analysis
When reference_text is provided, a separate CTC forced-alignment pass (Chapter 8)
produces per-letter timings, which are then fed into the madd analyser. The analyser
returns errors[] and an auto-calibrated haraka_ms .
Phase 5 — Result Assembly
The transcription, conﬁdence, elapsed time, and (optionally) madd errors are seri-
alised as JSON and returned to the Go server, which wraps it in the standard
{"status":"success","data":{...}} envelope.

8. Madd (Tajweed) Analysis
8.1Goal
Given a reference text the reciter is expected to say (Warsh orthography) and the au-
dio they actually produced, the madd analyser detects whether each expected vowel-
elongation ( madd ) was held for the correct number of counts ( harakat ), grading mistakes
asminor ormajor . It targets the Warsh ‘an Naﬁ’ riwaya via Tariq al-Azraq.
8.2Rules Covered
Table 8.1: Madd rules with expected counts and acceptance windows
Rule (key) Description Expected Window
tabii Natural madd (no hamza, no sukun) 2 1.5–2.7
badal Hamza precedes madd letter (qasr) 2 1.5–2.7
muttasil Madd + hamza, same word 6 5.0–7.5
munfasil Madd at end of word, hamza next 6 5.0–7.5
lazim_muthaqqal Madd + shaddah 6 5.0–7.5
lazim_mukhaffaf Madd + permanent sukun 6 5.0–7.5
aarid Madd near end of word at waqf (qasr) 2 1.5–2.7
lin Lin letter at waqf (qasr) 2 1.5–2.7
Warsh/Azraq permits multiple wujuh (2 / 4 / 6 counts) for Badal, Aarid, and Lin; the
analyser locks to the canonical qasr (2 counts) so each recitation is graded against a single
speciﬁc wajh. The acceptance window widens for the 6-count rules to absorb measurement
noise.

8.3Pipeline
Step 1 — Reference Parsing
parse_reference(text) inmadd.py walks each word character-by-character, detects
every madd letter (alif, waw, ya, dagger-alif, small waw/ya, alif-madda), classiﬁes its
hukm using vowel and post-letter context, and emits a MaddSpan carrying the expected
counts and tolerance.
Step 2 — Word Alignment
The reference words are aligned to ASR words via Levenshtein distance over diacritic-
stripped letters ( strip_diacritics also normalises alif-wasl, alif-maqsura, and ta-
marbuta to the forms the model emits). Reference words that the ASR didn’t recognise
emit a major omission error.
Step 3 — CTC Forced Alignment
madd_alignment.align_reference runs the model’s CTC head over the audio (one
extra forward pass, no RNNT cost), tokenises the trimmed reference with the model’s
SentencePiece tokenizer, and forced-aligns via torchaudio.functional.forced_align .
Output is a list of CharSpan(global_char_index, char, start, end) giving real per-
letter timings.
Step 4 — Haraka Calibration
estimate_haraka_ms computes one haraka in milliseconds from the recording itself. The
primary signal is total audio duration divided by total emitted characters (a speech-rate
proxy); two corroborating medians (all-char durations and Tabii-madd durations divided
by 2) are logged but not blended. The result is clamped to [120,600]ms.
Step 5 — Grading
For each MaddSpan , the analyser intersects its character range with the CTC spans (or
falls back to linear interpolation across the matched ASR word), measures the duration
in milliseconds, divides by haraka_ms to get observed counts, and grades against the
rule’s window: outside the window →major ; inside but |∆|>0.7from expected →
minor ; otherwise no error.
8.4Waqf Detection
A word is treated as a waqf (pause) when followed by ≥300ms of silence in the ASR
alignment, or when it is the last word in the transcript. Ayah boundaries are deliberately
72

notused — only acoustic silence drives the decision. aarid andlinspans are conditional
on waqf; a word-ﬁnal madd letter emits both an aarid candidate (requires waqf) and a
tabii fallback (skip-if-waqf), and exactly one of the two ever grades.
8.5Output Schema
Each detected mistake is a TajweedError JSON object containing the word, its index,
the rule ( Al-Madd ), the Arabic sub-rule name, severity ( minor /major ), expected and
detected counts, a one-sentence explanation, and a millisecond timestamp range. The full
payload also surfaces haraka_ms so the client can display the calibrated value.
