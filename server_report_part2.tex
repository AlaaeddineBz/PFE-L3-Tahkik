\chapter{Introduction}

This technical report documents the server-side infrastructure of the Tahkik project --- an AI-powered Quranic recitation analysis system specialising in the Warsh `an Nafi' riwaya (Tariq al-Azraq). The server bridges the Flutter mobile application with the speech-recognition model, accepts audio uploads (or live PCM streams), runs an optional Tajweed (madd) analysis against a reference text, and returns structured transcription and error results.

The production stack is composed of three containerised components running side-by-side on a single Linux host: a \textbf{Caddy} reverse proxy (auto-TLS via Let's Encrypt), a \textbf{Go} HTTP API gateway, and a \textbf{Python FastAPI} inference server backed by the NVIDIA NeMo \texttt{FastConformer Hybrid CTC/Transducer} model (\texttt{nvidia/stt\_ar\_fastconformer\_hybrid\_large\_pcd\_v1.0}). A WebSocket streaming endpoint (\texttt{/ws/stream}) delivers real-time partial transcription, and a separate \texttt{/api/v1/system-health} endpoint (plus its WebSocket variant) exposes host metrics, container status, and inference latency for the operations dashboard.

This report covers the architecture, component details, data pipeline, madd analyser, API contracts, deployment procedure, security model, and operational concerns of the server.

The Tahkik server runs as a three-container Docker Compose stack. Caddy terminates TLS and routes traffic; the Go API server handles HTTP REST traffic and system-health aggregation; the Python container runs the NeMo FastConformer model and also serves the live WebSocket stream. The Flutter client never speaks to the Python container directly --- it talks to the public hostname which Caddy proxies appropriately.

\section{Design Justification}

The architectural decisions were driven by the following constraints:

\begin{itemize}
    \item \textbf{Reverse proxy at the edge:} Caddy handles automatic TLS issuance from Let's Encrypt, CORS preflight for the admin dashboard, and routing rules (\texttt{/api/v1/*} and \texttt{/health} to the Go server; \texttt{/ws/*} straight to the Python container). This isolates concerns and lets the Go layer focus on JSON, not TLS.
    \item \textbf{Go for the API gateway:} Go's standard library \texttt{net/http} delivers a production-ready server with minimal dependencies, fast static-linked builds, and a tiny distroless image (\texttt{gcr.io/distroless/static-debian12:nonroot}, ${\sim}10$\,MB). The single external dependency (\texttt{github.com/coder/websocket}) is used only by the system-health stream.
    \item \textbf{NeMo FastConformer for inference:} The NVIDIA Arabic ASR checkpoint achieves 6.55\,\% WER on the EveryAyah Quran benchmark and ships a hybrid CTC + RNNT head. The RNNT head is used for transcription; the CTC head is reused for forced alignment against a reference text (see Chapter~\ref{chap:madd}).
    \item \textbf{Single-host deployment:} The full stack runs on one DigitalOcean droplet provisioned by \texttt{scripts/provision-droplet.sh}. CPU-only droplets work out of the box; a GPU droplet is enabled by a one-line overlay (\texttt{docker-compose.gpu.yml}) which sets \texttt{NEMO\_DEVICE=cuda} and reserves the NVIDIA device.
    \item \textbf{WebSocket streaming:} The \texttt{/ws/stream} endpoint enables real-time partial transcription with an 8-second sliding window, giving the reciter immediate feedback without waiting for the recording to finish.
\end{itemize}

\chapter{Component Details}

\section{Caddy Reverse Proxy}
\label{sec:caddy}

\subsection{Overview}
Caddy is the only container exposed to the public internet. It listens on ports 80 and 443, automatically issues a TLS certificate from Let's Encrypt on the first HTTPS request, and routes incoming traffic to the appropriate backend container.

\subsection{Routing Rules}
The full \texttt{Caddyfile} matches three classes of request:

\begin{lstlisting}[caption={Caddy routing (excerpt from Caddyfile)}]
@api_preflight { method OPTIONS  path /api/v1/* /health }
@api           { path /api/v1/* /health }
@ws            { path /ws/* }

handle @api { reverse_proxy go-server:8080 }
handle @ws  { reverse_proxy python:7860 }
\end{lstlisting}

The OPTIONS matcher injects the CORS headers (\texttt{Access-Control-Allow-Origin: \$DASHBOARD\_ORIGIN}) so the dashboard can call the API from a browser. WebSocket frames bypass the Go layer entirely.

\section{Go API Server}
\label{sec:go_api_server}

\subsection{Overview}
The Go API server is the entry point for all REST traffic. It listens on port 8080 (configurable via \texttt{PORT}) and exposes four endpoints: \texttt{POST /api/v1/evaluate} for audio evaluation, \texttt{GET /api/v1/system-health} for aggregated metrics, \texttt{GET /api/v1/system-health/ws} for the real-time stream, and \texttt{GET /health} for liveness checks. A fifth \texttt{healthcheck} sub-command exists so the distroless Docker image can run its own \texttt{HEALTHCHECK} without needing \texttt{curl}.

\subsection{Technology}
\begin{infobox}[title=Technology Stack]
\begin{itemize}
  \item \textbf{Language}: Go 1.26.2
  \item \textbf{Framework}: Standard library (\texttt{net/http})
  \item \textbf{WebSocket}: \texttt{github.com/coder/websocket} v1.8.14
  \item \textbf{Image}: \texttt{gcr.io/distroless/static-debian12:nonroot} (statically linked, ${\sim}10$\,MB)
\end{itemize}
\end{infobox}

\subsection{Key Implementation Details}

The server uses Go's built-in \texttt{http.NewServeMux} for routing. The \texttt{Evaluate} handler validates the uploaded audio file (max 50\,MB, supported formats: \texttt{.wav}, \texttt{.mp3}, \texttt{.m4a}, \texttt{.flac}, \texttt{.ogg}), reads the binary data, optionally captures a \texttt{reference\_text} form field, and delegates to the ML inference client.

\begin{lstlisting}[language=Go, caption={API entry point (cmd/api/main.go)}]
func main() {
    if len(os.Args) > 1 && os.Args[1] == "healthcheck" {
        healthcheckMain()
        return
    }
    port := os.Getenv("PORT")
    if port == "" { port = "8080" }

    log.Printf("inference backend: %s", ml.InferenceServerURL())
    log.Printf("version: git_sha=%s deployed_at=%s",
        handlers.GitSHA, handlers.DeployedAt)

    mux := http.NewServeMux()
    mux.HandleFunc("/api/v1/evaluate",         handlers.Evaluate)
    mux.HandleFunc("/api/v1/system-health",    requireBearer(handlers.SystemHealth))
    mux.HandleFunc("/api/v1/system-health/ws", handlers.SystemHealthWS)
    mux.HandleFunc("/health", func(w http.ResponseWriter, r *http.Request) {
        w.Header().Set("Content-Type", "application/json")
        _, _ = w.Write([]byte(`{"status":"ok"}`))
    })
    log.Printf("Tahkik server listening on :%s", port)
    http.ListenAndServe(":"+port, mux)
}
\end{lstlisting}

\subsection{Inference Client}

The \texttt{internal/ml/inference.go} module forwards audio to the Python backend as \texttt{multipart/form-data}, optionally including a \texttt{reference\_text} field. When that field is present the inference server runs madd analysis and the returned \texttt{Result} includes \texttt{errors[]} and \texttt{haraka\_ms} alongside the transcription.

\begin{lstlisting}[language=Go, caption={Inference result structure}]
type Result struct {
    Transcription    string         `json:"transcription"`
    ConfidenceScore  float64        `json:"confidence_score"`
    ProcessingTimeMs int            `json:"processing_time_ms"`
    HarakaMs         int            `json:"haraka_ms,omitempty"`
    Errors           []TajweedError `json:"errors,omitempty"`
}

type TajweedError struct {
    Word        string          `json:"word"`
    WordIndex   int             `json:"word_index"`
    Rule        string          `json:"rule"`
    SubRule     string          `json:"sub_rule"`
    Severity    string          `json:"severity"`
    Expected    string          `json:"expected"`
    Detected    string          `json:"detected"`
    Explanation string          `json:"explanation"`
    Timestamp   *AudioTimestamp `json:"timestamp,omitempty"`
}
\end{lstlisting}

Key characteristics:
\begin{itemize}
    \item HTTP client timeout: 5 minutes (accommodates large audio files and NeMo's RNNT decoder latency on CPU)
    \item Sends \texttt{Authorization: Bearer \{HF\_TOKEN\}} header when the env var is set
    \item Parses error responses from both \texttt{detail} (FastAPI) and \texttt{error} (legacy) JSON fields
\end{itemize}

\subsection{System-Health Aggregator}
\label{subsec:system_health}

\texttt{GET /api/v1/system-health} returns a single JSON blob composed from three sources, cached for 5\,s to avoid thrashing \texttt{/proc} and the Docker socket:

\begin{itemize}
  \item \textbf{Host metrics} --- read from \texttt{/host/proc/*} (uptime, loadavg, meminfo, cpu stat) and \texttt{statfs(/host)}. The compose file bind-mounts the host root read-only at \texttt{/host} so the Go container observes \emph{real} host values instead of cgroup-virtualised ones.
  \item \textbf{Container status} --- enumerated via the Docker Engine API over a read-only bind-mount of \texttt{/var/run/docker.sock}. Containers are filtered by the \texttt{com.docker.compose.project} label.
  \item \textbf{Inference + model snapshot} --- fetched from \texttt{GET python:7860/health}, which returns request counters, p50/p95 latency, and the loaded model identifier.
\end{itemize}

Bearer-token auth is enforced by the \texttt{requireBearer} middleware against the \texttt{HEALTH\_TOKEN} environment variable. If \texttt{HEALTH\_TOKEN} is unset the endpoint fails closed with \texttt{503 Service Unavailable} rather than going silently public. The WebSocket variant (\texttt{/api/v1/system-health/ws}) pushes a fresh snapshot every 2\,s; auth is via a \texttt{?token=} query parameter since browsers cannot set custom headers on a WebSocket handshake.

\section{Python Inference Server}
\label{sec:python_inference}

\subsection{Overview}
The Python container loads the NeMo FastConformer model once at startup and serves three endpoints on port 7860:

\begin{itemize}
  \item \texttt{GET /health} --- liveness + model + metrics snapshot
  \item \texttt{POST /evaluate} --- batch transcription (with optional madd analysis)
  \item \texttt{WS /ws/stream} --- real-time streaming transcription
\end{itemize}

\subsection{Technology}
\begin{infobox}[title=Technology Stack]
\begin{itemize}
  \item \textbf{Language}: Python 3.10
  \item \textbf{Framework}: FastAPI + Uvicorn
  \item \textbf{ML runtime}: NVIDIA NeMo (\texttt{nemo\_toolkit[asr]} 2.0.0) on PyTorch ${\geq}$\,2.2
  \item \textbf{Model}: \texttt{nvidia/stt\_ar\_fastconformer\_hybrid\_large\_pcd\_v1.0} \\(${\sim}115$\,M params, Hybrid CTC/RNNT)

\end{itemize}

\subsection{Model Loading}
The server prints \texttt{[inference] model ready (warmup <ms>)} when the model is fully loaded, which the system-health endpoint then exposes via the \texttt{model.loaded} boolean. Because NeMo 2.x's \texttt{transcribe()} cleanup path calls \texttt{encoder.unfreeze(partial=True)} --- which raises if the module was never frozen --- the model is explicitly frozen after \texttt{eval()}.

\begin{lstlisting}[language=Python, caption={Model loading at module import}]
MODEL_ID    = os.environ.get(
    "MODEL_ID",
    "nvidia/stt_ar_fastconformer_hybrid_large_pcd_v1.0",
)
NEMO_DEVICE = os.environ.get("NEMO_DEVICE", "cpu").lower()

import nemo.collections.asr as nemo_asr

model = nemo_asr.models.EncDecHybridRNNTCTCBPEModel.from_pretrained(
    model_name=MODEL_ID,
)
model = model.to(NEMO_DEVICE)
model.eval()
model.freeze()   # required for NeMo 2.x cleanup
\end{lstlisting}

\subsection{Inference Wrapper and Confidence Scoring}
\texttt{\_transcribe\_array(audio, *, with\_timestamps=False)} centralises the call into NeMo and handles all three possible return shapes of \texttt{transcribe()} (tuple of lists, list of \texttt{Hypothesis}, or list of strings). Confidence is derived from \texttt{Hypothesis.score} (an RNNT log-prob sum) via a soft exponential clamp so it lands in $[0, 1]$:

\begin{lstlisting}[language=Python, caption={RNNT score \texttt{$\rightarrow$} confidence mapping}]
def _score_to_confidence(score) -> float:
    if score is None:
        return 1.0
    return float(np.clip(np.exp(min(0.0, float(score) / 50.0)), 0.0, 1.0))
\end{lstlisting}

\subsection{Inference Concurrency Lock}
A module-level \texttt{asyncio.Semaphore(1)} caps total concurrent inferences across all connections to avoid CPU over-subscription on small droplets. Each WebSocket connection additionally holds a per-connection \texttt{asyncio.Lock} so a single client's partial-window tasks serialise cleanly.

\begin{emeraldbox}[title=Streaming Implementation Highlights]
\begin{itemize}
  \item \textbf{Sliding window}: Each partial transcribes the last \texttt{PARTIAL\_WINDOW\_S = 8} seconds of audio as one contiguous window, avoiding word-cutting at chunk boundaries.
  \item \textbf{Final window}: On \texttt{stop}, the last \texttt{FINAL\_WINDOW\_S = 60} seconds (NeMo handles much longer audio than Whisper's 30\,s) are re-transcribed for the final, higher-confidence result.
  \item \textbf{Per-connection + global locks}: One inference at a time inside a connection; one at a time across all connections.
  \item \textbf{Generation counter}: Each \texttt{reset} increments a counter; in-flight tasks compare their generation before emitting and discard stale output silently.
  \item \textbf{Silence drop}: After 10\,s of pure silence the audio buffer is dropped to prevent unbounded memory growth.
  \item \textbf{Exception isolation}: Errors inside partial-window tasks are logged with full traceback and do not tear down the WebSocket connection.
\end{itemize}
\end{emeraldbox}

\chapter{Data Pipeline}

\section{Audio Processing Flow}

When an audio file arrives at the server, it passes through the following stages:

\phaseblock{Phase 1 --- Audio Reception}{The Go server receives the multipart upload, validates the file extension and size (max 50\,MB), reads the binary payload into memory, optionally extracts the \texttt{reference\_text} form field, and forwards both to the Python backend.}

\phaseblock{Phase 2 --- Audio Preprocessing}{The Python backend writes the bytes to a temp file under \texttt{/tmp}, loads it via \texttt{librosa.load(sr=16000, mono=True)}, and clamps the duration to the last 60 seconds. Unlike Whisper, NeMo FastConformer handles long audio directly, so the legacy 30-second chunking step is no longer required.}

\phaseblock{Phase 3 --- Inference}{The audio array is handed to \texttt{model.transcribe(\dots)}. When a \texttt{reference\_text} is present, the call is invoked with \texttt{return\_hypotheses=True, timestamps=True} so the wrapper can extract per-word and per-character timestamps from the returned \texttt{Hypothesis} object.}

\phaseblock{Phase 4 --- Optional Madd Analysis}{When \texttt{reference\_text} is provided, a separate CTC forced-alignment pass (Chapter~\ref{chap:madd}) produces per-letter timings, which are then fed into the madd analyser. The analyser returns \texttt{errors[]} and an auto-calibrated \texttt{haraka\_ms}.}

\phaseblock{Phase 5 --- Result Assembly}{The transcription, confidence, elapsed time, and (optionally) madd errors are serialised as JSON and returned to the Go server, which wraps it in the standard \texttt{\{"status":"success","data":\{\dots\}\}} envelope.}

\section{PCM Conversion (WebSocket Path)}

The streaming path skips \texttt{librosa} entirely. Raw PCM bytes are converted to float32 in-place:

\begin{lstlisting}[language=Python, caption={PCM 16-bit signed LE \texttt{$\rightarrow$} float32 [-1, 1]}]
def _pcm_bytes_to_float32(pcm_bytes: bytes) -> np.ndarray:
    int16_array = np.frombuffer(pcm_bytes, dtype=np.int16)
    return int16_array.astype(np.float32) / 32768.0
\end{lstlisting}

A simple RMS-based silence detector (\texttt{SILENCE\_THRESHOLD = 0.02}) skips inference on quiet windows and triggers the buffer-drop path after sustained silence.

\section{Confidence Scoring}

\begin{infobox}[title=Confidence Calculation]
The NeMo RNNT decoder returns a \texttt{Hypothesis.score} that is the sum of log-probabilities over the emitted token sequence (a negative scalar). The server converts it to a $[0, 1]$ confidence via:
$$\mathrm{conf} = \mathrm{clip}\!\left(\exp\!\left(\min\!\left(0,\,\frac{\text{score}}{50}\right)\right),\, 0,\, 1\right)$$
The divisor and clamp were tuned so typical correctly-recited ayahs score above 0.9 and short, low-confidence partials still fall comfortably below 1.0. Values are monotonic but not directly comparable to the Whisper-era \texttt{avg\_logprob}-based confidence.
\end{infobox}

\chapter{Madd (Tajweed) Analysis}
\label{chap:madd}

\section{Goal}
Given a reference text the reciter is expected to say (Warsh orthography) and the audio they actually produced, the madd analyser detects whether each expected vowel-elongation (\emph{madd}) was held for the correct number of counts (\emph{harakat}), grading mistakes as \texttt{minor} or \texttt{major}. It targets the Warsh `an Nafi' riwaya via Tariq al-Azraq.

\section{Rules Covered}

\begin{table}[H]
  \centering
  \caption{Madd rules with expected counts and acceptance windows}
  \label{tab:madd_rules}
  \renewcommand{\arraystretch}{1.3}
  \begin{tabular}{llcc}
    \toprule
    \textbf{Rule (key)} & \textbf{Description} & \textbf{Expected} & \textbf{Window} \\
    \midrule
    \texttt{tabii}           & Natural madd (no hamza, no sukun)  & 2 & 1.5--2.7 \\
    \texttt{badal}           & Hamza precedes madd letter (qasr)  & 2 & 1.5--2.7 \\
    \texttt{muttasil}        & Madd + hamza, same word            & 6 & 5.0--7.5 \\
    \texttt{munfasil}        & Madd at end of word, hamza next    & 6 & 5.0--7.5 \\
    \texttt{lazim\_muthaqqal}& Madd + shaddah                     & 6 & 5.0--7.5 \\
    \texttt{lazim\_mukhaffaf}& Madd + permanent sukun             & 6 & 5.0--7.5 \\
    \texttt{aarid}           & Madd near end of word at waqf (qasr) & 2 & 1.5--2.7 \\
    \texttt{lin}             & Lin letter at waqf (qasr)          & 2 & 1.5--2.7 \\
    \bottomrule
  \end{tabular}
\end{table}

Warsh/Azraq permits multiple \emph{wujuh} (2 / 4 / 6 counts) for Badal, Aarid, and Lin; the analyser locks to the canonical \emph{qasr} (2 counts) so each recitation is graded against a single specific wajh. The acceptance window widens for the 6-count rules to absorb measurement noise.

\section{Pipeline}

\phaseblock{Step 1 --- Reference Parsing}{\texttt{parse\_reference(text)} in \texttt{madd.py} walks each word character-by-character, detects every madd letter (alif, waw, ya, dagger-alif, small waw/ya, alif-madda), classifies its hukm using vowel and post-letter context, and emits a \texttt{MaddSpan} carrying the expected counts and tolerance.}

\phaseblock{Step 2 --- Word Alignment}{The reference words are aligned to ASR words via Levenshtein distance over diacritic-stripped letters (\texttt{strip\_diacritics} also normalises alif-wasl, alif-maqsura, and ta-marbuta to the forms the model emits). Reference words that the ASR didn't recognise emit a \texttt{major} \emph{omission} error.}

\phaseblock{Step 3 --- CTC Forced Alignment}{\texttt{madd\_alignment.align\_reference} runs the model's CTC head over the audio (one extra forward pass, no RNNT cost), tokenises the trimmed reference with the model's SentencePiece tokenizer, and forced-aligns via \texttt{torchaudio.functional.forced\_align}. Output is a list of \texttt{CharSpan(global\_char\_index, char, start, end)} giving real per-letter timings.}

\phaseblock{Step 4 --- Haraka Calibration}{\texttt{estimate\_haraka\_ms} computes one \emph{haraka} in milliseconds from the recording itself. The primary signal is total audio duration divided by total emitted characters (a speech-rate proxy); two corroborating medians (all-char durations and Tabii-madd durations divided by 2) are logged but not blended. The result is clamped to $[120, 600]$\,ms.}

\phaseblock{Step 5 --- Grading}{For each \texttt{MaddSpan}, the analyser intersects its character range with the CTC spans (or falls back to linear interpolation across the matched ASR word), measures the duration in milliseconds, divides by \texttt{haraka\_ms} to get observed counts, and grades against the rule's window: outside the window $\rightarrow$ \texttt{major}; inside but $|\Delta| > 0.7$ from expected $\rightarrow$ \texttt{minor}; otherwise no error.}

\section{Waqf Detection}

A word is treated as a \emph{waqf} (pause) when followed by ${\geq}\,300$\,ms of silence in the ASR alignment, or when it is the last word in the transcript. Ayah boundaries are deliberately \emph{not} used --- only acoustic silence drives the decision. \texttt{aarid} and \texttt{lin} spans are conditional on waqf; a word-final madd letter emits both an \texttt{aarid} candidate (requires waqf) and a \texttt{tabii} fallback (skip-if-waqf), and exactly one of the two ever grades.

\section{Output Schema}

Each detected mistake is a \texttt{TajweedError} JSON object containing the word, its index, the rule (\texttt{Al-Madd}), the Arabic sub-rule name, severity (\texttt{minor} / \texttt{major}), expected and detected counts, a one-sentence explanation, and a millisecond timestamp range. The full payload also surfaces \texttt{haraka\_ms} so the client can display the calibrated value.
